# FRONTEND TASKS (MVP)

## 1. Базовая инициализация
- Создать frontend (React + TypeScript).
- Подключить Telegram WebApp SDK.
- Настроить мобильный viewport и safe-area.
- Настроить роутинг экранов MVP.

## 2. Авторизация
- При старте получать `initData` из Telegram WebApp.
- Отправлять в `POST /auth/telegram`.
- Хранить JWT в памяти/безопасном storage.
- Реализовать guard приватных экранов.

## 3. Экраны
- `Splash/Init`.
- `Лобби`.
- `Игра`.
- `Результат партии`.
- `Профиль/Статистика`.
- `Рейтинг` (общий/суточный).
- `История партий`.

## 4. Лобби UI
- Кнопка `Найти игру`.
- Кнопка `Выйти из очереди`.
- Список ожидающих игроков по нику.
- Кнопка `Вызвать` у каждого игрока.
- Отображение входящих инвайтов (принять/отклонить).

## 5. Экран игры
- Доска 8x8 (адаптивная, touch-friendly).
- Выбор фигуры и отображение легальных ходов.
- Отправка `game:move` через WebSocket.
- Отображение очереди хода.
- Кнопки: `Предложить ничью`, `Сдаться`.
- Диалоги подтверждений действий.

## 6. Realtime-состояние
- Подключение socket после успешного auth.
- Обработка `match:found` -> переход на игру.
- Обработка `game:state` и `game:move:applied`.
- Обработка `game:finished` -> экран результата.
- Авто-reconnect и повторная синхронизация состояния.

## 7. Результат и реванш
- Показ результата и причины завершения партии.
- Кнопка `Реванш`.
- Отображение статуса согласия второго игрока.

## 8. Профиль и статистика
- Получение `GET /me`.
- Показ: победы, поражения, ничьи, всего игр, очки.

## 9. Рейтинг
- Экран `Общий рейтинг` (`GET /leaderboard/global`).
- Экран `Суточный рейтинг` (`GET /leaderboard/daily`).
- Подсветка текущего игрока.

## 10. История игр
- Список партий (`GET /history`).
- Карточка партии: соперник, цвет, результат, дата.
- Переход в детали партии (`GET /games/:id`).

## 11. UI/UX требования
- Только русский язык.
- Минималистичный стиль.
- Крупные touch targets.
- Плавные переходы между статусами игры.
- Явные состояния: загрузка, ошибка сети, пустой список.

## 12. Техническое качество
- Типизация API-ответов.
- Централизованный state (store/query cache).
- Обработка edge-cases при разрыве сети.
- Базовые e2e smoke тесты ключевых сценариев.
